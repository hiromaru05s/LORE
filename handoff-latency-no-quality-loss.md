# メモ：品質を落とさず速度だけ上げる（会話レイテンシ）

> 現状 1ターン 3〜6秒。原因はサンプリング/プロンプトではなく、**1ターンに LLM が直列2回（① scoreAnswer → ⑤ generate）＋ DB往復4回**走る構造。
> ここに挙げるのは**出力テキストの質を一切変えずに**秒数だけ削る手だけ。プロンプト・温度・controller のロジックは触らない。

---

## 効く順（品質中立）

### 1. ① score と ⑤ generate を並列化（最大・最優先）
- **今**：`scoreAnswer`(LLM) を待ってから `decideMove` → `generate`(LLM)。完全直列。
- **変更**：手の決定（`decideMove`）が依存する `contour.material` は**過去ターンからの累積**。今回ぶんの加点は閾値超えにわずかに寄与するだけなので、**今回の score を1ターン遅れで反映**しても strike タイミングはほぼ不変。
  - 手は「前ターンまでの状態」で決める → `generate` を即発射
  - `scoreAnswer` は**同時に**（`Promise.all`）走らせ、結果は `recordAnswer` に回して**次ターンの燃料**にする
- **効果**：直列2回 → 並列1回ぶん ≒ **体感ほぼ半減**。
- **品質**：生成プロンプトの中身は不変。strike が1ターン遅延しうるだけで、ペーシング閾値の範囲内＝体験は変わらない。
- **リスク**：低。`answer` アクションの制御フロー局所改修のみ。

### 2. ⑤ の message をストリーミング（体感を直接削る）
- **今**：`response_format: json_object` で全文生成を待ってから返す。
- **変更**：`message` フィールドだけ先にストリーム配信 → **初トークン ~1秒で出始める**。`choices`/`missCandidates` は後追いで充填。
- **効果**：実時間でなく**体感レイテンシ**を大きく削る（人は「出始め」で速さを判断する）。
- **品質**：最終テキストは同一。見え方が変わるだけ。
- **リスク**：中（FE側のストリーム受けとJSON分割配信の実装が要る）。Convex の streaming 対応方式に依存。

### 3. score スロットを実際に速いモデルへ
- **今**：`tuning.ts` で `modelPro`/`modelFlash` とも `deepseek-chat`（重いV3）。「Flash運用」だが実体は速くない。
- **変更**：採点（①）は判断材料を出すだけの軽タスク。**小型・低レイテンシモデル**に振る（DeepSeekの軽量枠 or 別プロバイダの高速モデル）。生成（⑤）は今のモデルのまま。
- **効果**：①の往復時間を短縮。1と併用で並列の「遅い方」が縮む＝さらに速く。
- **品質**：採点は3軸0〜3の粗い数値なので小型モデルで十分。**生成品質は不変**（生成モデルは据え置き）。
- **リスク**：低〜中（モデル選定とプロンプト微調整、採点の妥当性を eval で確認）。

### 4. 自明なターンは採点をスキップ
- **今**：tap反応・短い相槌でも毎回フル LLM 採点。
- **変更**：`isSubstantive` 未満が明らかな入力（極短文・定型tap）は①を**省略**し、`materialWeight=0` 相当で `decideMove` をヒューリスティック処理。
- **効果**：軽いターンで LLM 呼び出しが1回だけになる。
- **品質**：中身の薄い回答は元々 strike 燃料にならない＝採点しなくても判断は同じ。**生成は通常どおり**。
- **リスク**：低。スキップ条件を保守的に（文字数・inputMode で線引き）。

### 5. DB往復の前倒し・集約（細かいが純粋ゲイン）
- **今**：前段 `consumeBoundaryAnswer` → `recordAnswer` → `loadCtx` → ⑥ と DB往復が直列に4回。
- **変更**：`loadCtx` を①と**並列**で先読み（score を待たずに引ける部分＝session/contour/recentTurns/prefs は domain非依存で取れる）。`recordAnswer` の確定書き込みは generate と並走 or 後追い。
- **効果**：数百ms の積み上げを削る。
- **品質**：読み書きの順序整理のみ、出力に影響なし。
- **リスク**：低〜中（domain依存の読み出しだけ score 後に回す切り分けが要る）。

---

## やらないこと（品質が落ちる/効かない）
- temperature/top_p 等の変更 → **秒数に効かない**。
- `max_tokens` を絞って短くする → 速くなるが**返しの質・長さが変わる**＝品質変更なので対象外（別途 reply_length ダイヤルの話）。
- 候補生成 best-of-N の削減 → そもそも今は単発なので無関係。むしろ品質向上側の別件。
- プロンプト短縮で入力トークン削減 → 効果は小さく、ガードレールを削ると品質劣化リスク。

---

## 実装状況（2026-06-23）
- ✅ **1 並列化**：`sendTurn` で手を前ターン状態から決定し、`scoreAnswer` と `generate` を `Promise.all` で並走。採点は `recordAnswer` 経由で次ターンの燃料に。`loadCtx` は domain 省略時 `lastDomain` で先読み。
- ✅ **4 自明ターンの採点スキップ**：4文字以下は採点LLMを呼ばず材料0扱い（`isTrivialAnswer`）。
- ⬜ **2 ストリーミング**：FE↔Convex のストリーム受けが要るため未。体感への効きは最大なので次点。
- ⬜ **3 score高速モデル**：現状DeepSeek1系のみ。別枠/別プロバイダの高速モデル契約待ち（envで差すだけにはなっている）。
- ⬜ **5 DB整理**：1で loadCtx 先読み化は入れたが、recordAnswer の並走化までは未（影響小）。

## 着手順の推奨
**1（並列化）→ 3（score高速モデル）→ 4（採点スキップ）→ 2（ストリーミング）→ 5（DB整理）**

1 だけで体感ほぼ半減・最小改修。2 は実装重いが体感への効きが一番大きいので、1の後に余力があれば。いずれも**生成プロンプト・controller・サンプリングは無改変**＝出力テキストの質は保たれる。
