# ハンドオフ：会話設計 → LLM/FE/BE 実装タスク

> 別セッション（LLM/BE調整）向け。背景の原理は `LORE.md §3`（会話の質感を生む原理／preference 適応）が正。本書は**実装に落とすべき差分**だけを抽出。

---

## 0. 前提（1分で）

会話の質感「ゾワッ／テンポ／楽しい」は同じエンジン＝**ベースの予測可能性 × たまの逸脱**。普段は予測可能に保ち、稀に外す。コントラストが価値。

調整の核心ルール：**lore は当てる／preference は聞く。**
- **lore（どういう人か）** = 推定して strike で当てる。ここがゾワッ。聞いたら負け。
- **preference（どう話されたいか）** = 推定不能なので明示で聞く。format/consent の話なのでゾワッは削れない。

**送信ダイヤル ≠ 受信ダイヤル**：ユーザーが打つ量から、返してほしい量は推定不能（長文を読むのは好きだが短く打つ人が多い）。

---

## 1. LLM チューニング

プロンプト組み立て時に **preference state** を条件付けし、controller が「手」を選ぶ。調整すべきは以下7点。

1. **テンポ／逸脱の集中**
   - strike を安売りしない（substantive 2〜3 に対し 1）。`dig`/`reflect` は低密度・予測可能側、`strike` に逸脱を集中。
   - 応答長を**意図的に分散**（毎ターン同じ情報密度がテンポを壊す主因）。1語で返す勇気、相手のターンを全部回収しない、間を許す。`reflect`/`reask` は短打。

2. **言い当て強度ダイヤル（`strike_intensity`）**
   - `やんわり仮説`（「〜かも？」ヘッジ多め）⇄ `断定ズバッ`。ユーザー preference で切替。
   - これはゾワッの強さ＝コア体験の安全弁。プロンプトで断定/ヘッジの度合いを条件付け。

3. **受信ボリュームダイヤル（`reply_length`）**
   - 返す量・深さを送信長と**独立**に制御。short/long をプロンプト指示＋max_tokens で。
   - 送信側の長さは推定にも矯正にも使わない。

4. **トーン／口調・人格（`persona`）**
   - **ミラーリング禁止**：ユーザーが「お前」と言ったらAIも「お前」、はNG。文体の鏡写しをしない明示制約をシステムプロンプトに。
   - 口調・人格は会話の中で自然に設定された値を使う（初期はニュートラル、設定で上書き）。

5. **テーマ境界ゲーティング（`topic_boundaries`）**
   - NG 認定テーマには `dig`/`strike` で踏み込ませない。controller 側で候補をフィルタ＋プロンプトにも明記。
   - 内面に潜るアプリなので地雷を踏むと一発で気持ち悪い＝最優先の安全制約。

6. **コールドスタートの順序**
   - **ゾワッを先・設定提示を後**。初手で軽い read（strike寄り）を一発見せてから、話し方調整を案内。逆だと「設定を埋めるアプリ」化。

7. **miss 歓迎の規範**
   - 初回に一度、「違ったら教えて／君に興味がある」を宣言（下記コピー）。以降は外し回復フロー（既存 miss types）が機能する前提で攻めの strike を許容。

## 2. BE（Convex）追加開発

### 2-1. スキーマ：`preferences`（lore とは別物として保持）
preference は**メタデータであって lore fragment ではない**。内面モデルに混ぜない。

```
preferences (per user)
  replyLength       : 'short' | 'long' | null
  strikeIntensity   : 'soft'  | 'bold' | null
  persona           : { tone, register, ... } | null   # 会話から設定
  topicBoundaries   : [{ topic, state: 'allowed'|'blocked'|'unasked', source: 'asked'|'reaction' }]
  elicitation       : { askedTopics:[], lastAskedAt, nextEligibleAt }  # 漸進・再訪の制御
```

### 2-2. preference エリシテーション controller（新規）
**いつ聞くか**を判断するロジック。会話中に何度も聞かない。
- コールドスタート：一度だけチュートリアル的に提示 → 以降は会話から消す。
- **漸進**：テーマ境界は一度に数十カテゴリを出さない。**3つずつ**提示 → 全部OKならその許可文脈から会話継続 → `nextEligibleAt` で間隔を空けて再訪。
- **反応起点**：話題を急に切り上げ等のシグナル検知時に、追い討ちで「嫌なら次回から聞かないから教えて」を1回だけ。

### 2-3. シグナル検知（新規・軽め）
- **急な話題切り上げ／離脱**（短返信＋話題転換、トーン低下など）を検知 → テーマ境界の確認をトリガ。
- 受信preference は送信行動から推定**しない**（誤爆するため）。検知は境界トリガ用途に限定。

### 2-4. オーケストレーション差分（`conversation.ts`）
- プロンプト組み立てに preference state を注入（intensity / length / persona / boundaries）。
- `dig`/`strike` の候補生成を `topicBoundaries` でフィルタ。
- 設定値を受信ダイヤルの初期値として読む。override は設定起点で。

## 3. FE（`LORE.dc.html` / `lore-convex.js`）追加開発

### 3-1. 会話内マイクロ調整 UI（tap choices）
チュートリアル的に**一度だけ**自然に出す。フォームではなく会話の一手。
- 言い当て強度：「ズバリ言っちゃっていい？／控えめにする？」を tap で。
- テーマ境界：**3つずつ**チップで提示、OK/NG。漸進・間隔再訪を BE の `elicitation` に従って出す。
- トーン/口調：会話から拾った値の確認（ミラーリングはしない）。
- 反応起点トリガ時の「次回から聞かない？」軽アクション。

### 3-2. 設定サーフェス（軽トグル 1〜2個）
- replyLength / strikeIntensity / persona / topicBoundaries を後から変更可能に。
- **フォーム化しない**（診断アプリ臭＝原則7回避）。「いつでもここで変えてね」と一度誘導 → 以降ユーザー起点。

### 3-3. コピー（初回の宣言系）
- miss 歓迎：「君のことを知りたいから色々聞くし『こうじゃない？』とも言うけど、違ったらその度に教えてね。僕は君にすごく興味があるんだ。」
- 強度確認：「僕のミッションは君を知ることだから、あえて聞くよ。君について思うこと、結構ズバリ言っちゃっていい？ それともちょっと控えめに伝えようか？」

### 3-4. 入力側の逃げ道（既存強化）
- 短く打ちたい人を罰しない：`tap`/短文の選択肢を常時用意（送信長は矯正しない）。

## 4. やらないこと（明示ガード）

- **送信長から受信preferenceを推定する** → 無相関。やらない。
- **性格・価値観・本音を聞く** → それは当てる対象（lore）。聞いたら負け。
- **文体のミラーリング** → ウザい。禁止。
- **受信View（相手にどう見えるか）の初回プレビュー** → 不要。自分が見る画面＝相手が見る画面で意図的に一貫済み。
- **preference を会話中に繰り返し聞く** → 一度→設定に逃がす。
- 通知・フォロー（既存の §8 方針どおり作らない）。
